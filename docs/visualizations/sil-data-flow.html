SIL Data Flow: CRRCSim ↔ smol Flight Controller

Created: 2026-03-20
Branch: main
Commit: 337a7b0a — Fix build errors, thread safety, and clean shutdown

Summary

CRRCSim (a 6-DOF flight dynamics simulator) and smol (a flight controller) communicate over UDP localhost:9002 using a lockstep binary protocol. The sim produces sensor data from its physics engine, the flight controller consumes it to run EKF + control loops, and sends back actuator commands that drive the simulated airplane.

The protocol is defined once in sil_protocol.h (shared between both repos). Each message carries a 4-byte header identifying its type. The lockstep is driven by the IMU message — both sides block until the other responds, creating a deterministic simulation with no wall-clock dependency.

An alternative mode exists: the joystick_passthrough Python tool can replace smol in the loop, sending actuator commands directly from a physical joystick. This lets you fly the airplane manually without the flight controller.

Detailed Description

sil_protocol.h (shared between both repos)

Defines 8 packed binary message structs and a 4-byte header. All structs use #pragma pack(push, 1) for wire-exact layout. static_asserts enforce sizes at compile time. A companion sil_protocol.py parses this header at import time to generate matching Python ctypes structs — single source of truth, zero duplication.

inputdev_apm.cpp (CRRCSim side)

The sim's input device plugin. Called once per physics step via getInputData() from CRRCSim's main loop. When protocol="smol" is set in XML config, it takes the smol path:

  1. Block-read a SimActuatorMsg from the flight controller (via charDevice->read()). This is the lockstep synchronization point — the sim waits here until the FC responds.
  2. Map the actuator's aileron/elevator/throttle/rudder to TSimInputs which drives the FDM.
  3. Extract FDM state from Global::aircraft->getFDM(). All positions/velocities are converted from feet (CRRCSim internal) to meters (SI). Body-frame accelerations include gravity subtraction then world-to-body rotation.
  4. Send sensor messages at decimated rates using _cycle counter: IMU + Truth + RC every cycle, Mag every 2nd, Baro every 4th, Airspeed every 10th, GPS every 20th. Each sensor has configurable Gaussian white noise added (from XML <noise> element). Magnetometer uses WMM2015 earth field rotated to body frame via DCM.

Uses a static std::mt19937 rng(42) — deterministic seed for reproducible noise sequences across runs.

apm.h / apm.cpp (CRRCSim side)

Thin wrapper around BufferedCharDevice. Provides send_msg(void*, len) and recv_actuator(SimActuatorMsg*) which delegate directly to charDevice->write() / charDevice->read(). The underlying transport is configured by the XML device string (default: udpserver,127.0.0.1/0.0.0.0,9002).

hal_native_sil.cpp (smol side)

The SIL HAL implementation. All state is file-static (no classes). Key behaviors:

  • ReadImu() — The lockstep entry point. Does a blocking recvfrom() until a SimImuMsg arrives. Then drains all remaining pending messages with MSG_DONTWAIT, caching each by type and setting per-sensor new-data flags. Returns an ImuSample. This is what makes the flight loop block until the sim sends data.
  • ReadGps/Mag/Baro/Rc() — Check-and-clear: if the new-data flag is set, return the cached data and clear the flag. Otherwise return nullopt. This mirrors how the real hardware HAL works with different sensor rates.
  • WriteServos() — Converts PWM µs back to normalized floats using servo params, builds a SimActuatorMsg echoing the last IMU sequence number, and sends via sendto().
  • Clock — Driven by SimImuMsg.time_sec, not wall clock. SleepUntilNextCycle() is a no-op.

main_native_sil.cpp (smol side)

Entry point for the SIL build. Parses --output <path> for snapshot logging. Initializes HAL and FlightLoop, spawns a background thread (log drain + telemetry), then loops FlightLoop::Step() until SIGINT or negative time from ReadImu().

flight_loop.cpp (smol side, unmodified)

The 100 Hz control loop. Each Step(): read all sensors → EKF time update (IMU) → EKF measurement updates (GPS/Mag/Baro/Accel) → get state estimate → parse RC input → cascade controller → servo mixer → hal::WriteServos(). The flight loop has no knowledge of SIL vs hardware — it calls the same hal::* API either way.

joystick_passthrough/main.py (Python tool)

Drop-in replacement for smol in the lockstep loop. Reads a physical joystick via pygame, maps axes through a YAML calibration config, and sends SimActuatorMsg to the sim on each cycle. Blocks on recvfrom() waiting for SimImuMsg (same lockstep). Drains all other sensor messages without processing them.

Observations

  • Unit conversion duplication: getInputDataAPM() and getInputDataSmol() both extract FDM state with identical feet→meters conversion. Could be a shared helper, but the two paths diverge enough downstream that merging them would reduce readability.
  • Hardcoded channel mapping in WriteServos: hal_native_sil.cpp:337-340 assumes conventional airframe layout (ch0=aileron, ch1=elevator, ch2=throttle, ch3=rudder). This works because the servo mixer already handles airframe-specific mixing and outputs channels in this order for conventional layouts. But it would break if someone changed the mixer output channel order without updating the SIL HAL.
  • RC mode switches hardcoded: inputdev_apm.cpp:416-417 sends mode_a=2012, mode_b=2012 (both high = Manual mode). There is no way to change flight mode from the sim side currently.
